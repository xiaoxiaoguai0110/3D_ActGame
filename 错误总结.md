# 玩家自动旋转问题总结

## 问题描述

玩家角色在移动时会不受控制地**凭空自动旋转**，即使不按方向键也会自己转圈。

---

## 原因分析

角色动画文件（如 `run_forward.anim`、`walk_front.anim`）中**包含了旋转数据**，动画播放时会直接写入 `Transform.rotation`，导致 Animator 不断修改角色的朝向，与脚本控制的旋转产生冲突。

```
动画写入 Transform.rotation  →  角色自动旋转
脚本调用 rb.MoveRotation()   →  也在控制旋转
                ↓
        两方冲突 → 角色失控
```

---

## 解决方案

### 核心思路

通过 **Rigidbody 的 Freeze Rotation** 阻挡动画对旋转的修改，同时保留脚本的转向控制。

### 具体步骤

1. **在 Inspector 中冻结 Rigidbody 的 Y 轴旋转**
   - 选中角色 GameObject
   - 找到 Rigidbody 组件 → **Constraints**
   - 勾选 `Freeze Rotation Y`

2. **转向全部使用 `rb.MoveRotation()`**
   - 不要用 `transform.rotation` 直接赋值
   - 使用 `Rigidbody.MoveRotation()` 进行平滑旋转

### 原理说明

| 操作 | 方式 | 受 Freeze 影响？ |
|------|------|:---:|
| `rb.MoveRotation()` | 走 Rigidbody 内部通道 | ❌ 不受影响 |
| 动画修改 Transform | 直接写入 Transform | ✅ 被冻结阻止 |
| 物理碰撞反作用力 | 外部力 | ✅ 被冻结阻止 |

`rb.MoveRotation` 是 Rigidbody 自己的 API，不会绕过 Freeze 约束；而**动画、碰撞、其他脚本直接改 Transform** 的旋转，都会被 Freeze 挡住。

---

## 最佳实践

### Rigidbody 约束标准配置

```
冻结 X 轴 — 防止角色前后倾倒
冻结 Z 轴 — 防止角色左右歪斜
冻结 Y 轴 — 挡住动画/物理的幽灵旋转
        ↓
转向全部用 rb.MoveRotation()
```

### 代码示例

```csharp
// ✅ 正确：用 Rigidbody 控制旋转
private void FixedUpdate()
{
    if (direction.magnitude > 0.01f)
    {
        Quaternion targetRotation = Quaternion.LookRotation(direction);
        rb.MoveRotation(Quaternion.Slerp(rb.rotation, targetRotation, rotateSpeed * Time.fixedDeltaTime));
        rb.MovePosition(rb.position + transform.forward * moveSpeed * Time.fixedDeltaTime);
    }
}
```

---

## 总结

| 控制方式 | 能否冻结 Y 轴？ | 说明 |
|---------|:-----------:|------|
| `rb.MoveRotation` 控制转向 | ✅ 可以冻结 | MoveRotation 走自有通道，不受限制 |
| `transform.rotation` 控制转向 | ❌ 不能冻结 | 冻结后转向也会失效 |

遇到角色无故自动旋转，优先检查 **动画是否写入了旋转**，然后通过 **Freeze Rotation Y + MoveRotation** 解决。

# 动画过渡卡住问题总结

## 问题描述

停止移动时，walk 动画卡在当前帧不动，walk → idle 的过渡虽然触发了，但 walk 的最后一帧画面卡着，idle 在背后循环播放。

## 原因分析

`CurrentSpeed` 参数同时承担了两个职责：**过渡条件** 和 **动画播放速度**。

```
CurrentSpeed = direction.magnitude
       ↓
① 作为过渡条件 → CurrentSpeed < 0.1 时触发 walk → idle
② 作为动画速度 → walk 的 SpeedParameter 绑定了 CurrentSpeed

停止按键时：
  CurrentSpeed = 0
  作为条件 → 触发过渡 ✅
  作为速度 → walk 动画速度为 0，卡死 ❌
```

同一个参数既当"油门"又当"刹车"，停止时动画被自己冻住了。

## 解决方案

### 核心思路

`CurrentSpeed` 只负责过渡条件，不和动画播放速度绑定。

### 具体步骤

1. **去掉 walk 状态的 SpeedParameter 绑定**
   - 在 Animator 中选中 walk 状态
   - Inspector → 将 `Speed Parameter` 设为空（None）
   - 把 `Speed` 恢复为 1

2. **代码保持原样，只设置 CurrentSpeed**
   ```csharp
   animator.SetFloat("CurrentSpeed", direction.magnitude);
   ```

### 原理说明

| 职责 | 应该用谁控制？ |
|------|:------------:|
| 过渡条件（walk ↔ idle） | `CurrentSpeed` 参数 |
| 动画播放速度 | 状态自己的 `Speed` 属性，或单独的参数 |

**关键教训：** 一个 Animator 参数不要同时控制过渡条件和播放速度，否则会互相干扰。

---

## 总结

遇到动画卡住不动，优先检查 **SpeedParameter 是否绑定了和过渡条件相同的参数**。

如果需要让动画速度跟随移动速度（解决滑步），应该用**单独的参数**来控制，不要和过渡条件共用。

# Apply Root Motion 勾选后移动方向失控

## 问题描述

为了使用动画里的位移数据，勾选 Animator 的 **Apply Root Motion** 后，按 WASD 角色始终只朝一个方向走，摄像机旋转后方向也变不了。

不勾选时通过 `rb.MovePosition` 控制移动一切正常，但勾选后动画的 Root Motion 和脚本位移冲突，两个来源都在写位置。

## 原因分析

根源是**两股力量在抢位置**：

```
脚本 MovePosition  → 往 direction 方向走
动画 Root Motion   → 往动画自身的 forward 走（固定在世界方向）
         ↓
   两股力量互相覆盖 → 只有一个方向生效
```

更深层的原因是用 `transform.forward` 来算位移，但角色转向有 Slerp 延迟，`transform.forward` 还没转过去。而 Root Motion 又在此基础上叠加了自身位移，导致方向彻底混乱。

## 解决方案

### 核心思路

**把位移控制集中到 `OnAnimatorMove()` 中**，自己决定用脚本位移还是动画位移。

### 代码架构

```
Update()          → 只读输入 + 算方向 + 设动画参数（不写物理）
OnAnimatorMove()  → 统一接管所有位移
  ├─ 有 WASD 输入 → 脚本控制位移和旋转
  └─ 没输入       → animator.ApplyBuiltinRootMotion()
                    有 Root Motion 的动画 → 生效
                    idle 动画（无位移） → 原地不动
```

### 关键代码

```csharp
private void Update()
{
    // 读输入，算方向，设动画参数
    Vector3 dir = camRight * input.x + camForward * input.y;
    inputMagnitude = dir.magnitude;

    animator.SetFloat("CurrentSpeed", inputMagnitude);
    animator.SetBool("IsRun", running);

    if (inputMagnitude > 0.01f)
        targetMoveDir = dir.normalized;
    else
        targetMoveDir = Vector3.zero;
}

private void OnAnimatorMove()
{
    if (inputMagnitude > 0.01f)
    {
        // 脚本控制：转方向 + 位移
        rb.MoveRotation(Quaternion.Slerp(rb.rotation, targetRot, rotateSpeed * Time.deltaTime));
        rb.MovePosition(rb.position + targetMoveDir * speed * Time.deltaTime);
    }
    else
    {
        // 动画控制：Root Motion 生效
        animator.ApplyBuiltinRootMotion();
    }
}
```

### 为什么不能放在 FixedUpdate 里？

`OnAnimatorMove` 在 Animator 处理完动画状态之后、写入 Transform 之前调用，是 Unity 预留的**唯一可以安全拦截 root motion 的时机**。放在 FixedUpdate 里写位置，Animator 会在之后的 Update 循环中覆盖你的修改。

## 其他要点

- **Rigidbody 约束**：保持 **Freeze Rotation XYZ**，防止动画旋转覆盖脚本转向
- **方向来源**：用 `targetMoveDir`（即时算出的方向），不用 `transform.forward`（角色朝向有延迟）
- **Animator.applyRootMotion**：代码中设为 `true`，但通过 `OnAnimatorMove` 控制何时用 root motion
